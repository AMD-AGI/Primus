### 2.1 Training Log Patching Architecture (`training_log_patches.py`)
- **Problem**: Multiple features (`mlflow`, `rocm_mem`) needed to patch `megatron.training.training_log` simultaneously, leading to potential conflicts and code duplication.
- **Solution**: Implemented a **Chain of Responsibility** pattern using `contextlib.ExitStack`.
- **Implementation**:
    - Defined a `TrainingLogExtension` protocol.
    - Implemented `RocmMonitorExtension` as a Context Manager that temporarily swaps `print_rank_last`.
    - Created `patch_training_log_unified` wrapper that iterates through active extensions and applies them dynamically using `ExitStack`.
- **Safety**: Changed iteration tracking from fragile argument position parsing (index 4) to a robust internal counter (`self.call_count`).

### 2.2 ROCm Memory Monitoring Optimization
- **Performance**: Restricted expensive `rocm-smi` system calls to specific iterations.
- **Logic**:
    - **HIP Stats**: Always collected when extension is active (low overhead).
    - **ROCm SMI Stats**: Only collected if `use_rocm_mem_info` is globally True OR if the current iteration matches `use_rocm_mem_info_iters`.
- **Output**: Stats are **appended** to the log string rather than replacing specific tokens, preserving the original log format.
- **Scope**: The `print_rank_last` replacement is strictly scoped within the `training_log` execution, preventing side effects on other validation logs.
