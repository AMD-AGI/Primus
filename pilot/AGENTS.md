# AGENTS.md — Working agreement for agents operating in `pilot/`

This file is the canonical entry for any AI agent (Cursor / Claude Code / Codex / in-house harness) doing work inside `pilot/`. Read it before reading anything else.

---

## 1. Language

- **All git-tracked files under `pilot/` are written in English.** This includes `skills/**/*.md`, `prompts/**/*.md`, `tools/**/*.py` (docstrings + comments), `schemas/**/*.json` (descriptions), `agent/**`, `integrations/**`, and the root design docs.
- **No mixed-language files.** If a Chinese mirror is needed (e.g. `README.cn.md`), keep it in a separate file; never mix CN and EN inside one file.
- **The English version is the source of truth.** When CN and EN diverge, EN wins.
- Rationale: Skills are LLM prompt fuel. English maximizes cross-LLM compatibility (Claude / GPT / Gemini / Codex / Grok), aligns with industry convention (DeepSpeed / Megatron / vLLM), and avoids tokenizer waste.

## 2. Layer discipline

| Layer | What goes here | What does NOT |
|-------|----------------|---------------|
| `skills/` | Domain knowledge (Markdown). Rules, formulas, taxonomy. | Code, runtime state, agent SDK refs |
| `prompts/` | Role prompts (Markdown), framework-agnostic. | API formats (`input_schema`, `tool_choice`) |
| `tools/` | Business actions (Python), exposed via CLI / MCP boundary. | Agent-SDK imports (anthropic / cursor-client / openai / ...) |
| `schemas/` | Data contracts (JSON Schema source + Pydantic mirror). | Implementation logic |
| `state/` | Runtime artifacts (YAML / JSON). gitignored. | Source code, knowledge |
| `integrations/<framework>/` | Thin adapter per framework. **Only place agent SDKs may appear.** | Domain logic |
| `agent/` | Optional Python reference harness (fallback runtime). | Anything that should be framework-agnostic — that belongs in `tools/` |

## 3. Non-negotiables

1. **Pilot core (`skills/` + `prompts/` + `tools/` + `schemas/` + `state/`) does NOT import any agent SDK.** SDK deps (`anthropic`, `cursor-client`, `openai`, ...) live only under `integrations/<framework>/` (and optionally under `agent/`).
2. **Tools expose process / MCP boundaries**, not Python function boundaries. Every tool module must support `python -m pilot.tools.<module> <subcommand>`.
3. **State is the file system.** No in-memory shared state across stages. Cross-stage memory goes through `state/`.
4. **Subagent isolation is the framework's job, not Pilot's.** Pilot only specifies the boundary table (§13.2 of `README.md`) — `subagent.spawn` is a protocol abstraction, concrete impl injected by `integrations/<framework>/`.
5. **`SubagentResult.summary < 200 tokens`** — enforced by schema validation. Workers exceeding it are rejected.
6. **Every Pilot tool requires a `cluster.yaml`** (see §4 below). Tools must NOT discover cluster topology, spawn containers, ssh between nodes, or invoke `salloc`/`sbatch`. The runtime environment is a *prerequisite*, not a Pilot concern.

## 4. Universal tool input contract: `cluster.yaml`

Every command under `python -m pilot.tools.<module>` takes exactly one mandatory environmental input: a `cluster.yaml` file conforming to `schemas/cluster_config.schema.json`. It is resolved in this priority order:

1. `--cluster-config <path>` CLI argument (highest)
2. `PILOT_CLUSTER_CONFIG=<path>` environment variable
3. `./cluster.yaml` in the current working directory (lowest)

If none resolves, the tool exits with `failure.kind=CLUSTER` and a message instructing the caller to prepare one. **Tools never fall back to ad-hoc auto-discovery.**

### What `cluster.yaml` declares (and what it doesn't)

`cluster.yaml` declares **only** the runtime mode Pilot is operating inside:

| Mode | What the user prepared before invoking Pilot |
|------|----------------------------------------------|
| `single` | One container is already running on the local host with N GPUs visible. Pilot stays inside, runs torchrun with `nnodes=1`. |
| `slurm`  | A SLURM allocation already exists (`salloc -N k …` or running inside `sbatch`). The job id is recorded in `slurm.job_id`. Pilot attaches to it via `srun --jobid=<id>` to fan out to the allocated nodes. |

It does NOT declare: docker image, mounts, `--device` flags, network topology, ssh hosts, IB HCA names, k8s manifests, mpi rankfile, etc. **Anything not in the schema is a pre-Pilot prep step the user/SRE owns.**

### Lifecycle and validation

- **Author**: human / SRE / `pilot init-cluster` helper (TBD). Not auto-generated by tools.
- **Lifetime**: tied to the prepared environment. When the SLURM job ends or the container exits, the file is stale.
- **Validation**: every tool entry runs three fast-fail checks before touching state:
  1. Schema-valid against `cluster_config.schema.json`
  2. If `mode=slurm`: `scontrol show job <job_id>` returns `JobState=RUNNING` and `nnodes`/`nodelist` (if specified) match
  3. At least one GPU visible inside the current process (`rocm-smi` or `nvidia-smi`)

Failures here count as infrastructure errors, not tuning rounds.

### Why this is non-negotiable

Without a single declarative input, every tool would re-invent its own cluster discovery (env-var sniffing, ssh probing, slurm fallbacks). That fragments behaviour and makes Worker context unbounded. Pinning the contract to one tiny YAML keeps every tool's input surface identical and makes `rerun on the same allocation` trivial: copy the file.

## 7. When uncertain

- Default action: **read more Skill, write less code**. Pilot is knowledge-first.
- If Skill rules are ambiguous, prefer the conservative choice (smaller search step, tighter constraint, more explicit logging).
- Escalate to humans via `FailureReport(failure_kind=UNKNOWN)` rather than guess.
